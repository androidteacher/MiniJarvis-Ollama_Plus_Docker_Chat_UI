Context Window & Quantization

Memory limits, file compression, and the hardware you need to run local LLMs

Context Window & Quantization

Two concepts that determine how much a model can remember and how much space it takes up on your machine. Understanding these will help you pick the right model for your hardware.

💾 These concepts directly affect which models run on your hardware
6 Context Window — The Model’s Short-Term Memory

The Context Window is the maximum amount of information (in tokens) the model can “keep in its head” at any single moment.

Think of it like a rolling whiteboard. As you type a prompt and the model generates a response, that text fills up the board. Once the board is full, the model has to erase the oldest information at the top to make room for new words at the bottom.

  • 📄 Why it matters: If you ask a model to summarize a 50-page PDF but the model only has an 8,000-token context window (~15–20 pages), it will “forget” the beginning of the document by the time it reaches the end.

⚖️ The “Goldilocks” Problem:

  • 🟥 Too Small: The model loses track of the conversation or forgets instructions you gave it ten minutes ago.
  • 🟦 Too Large: The model requires significantly more RAM/VRAM to stay “attentive,” which can slow down response time considerably.
7 Quantization — Shrinking the Giant

By default, the numbers (weights) in a raw AI model are stored as high-precision decimals (usually 16-bit or 32-bit). This makes files massive — a raw 7B parameter model takes up 14–28 GB.

Quantization is the process of “rounding” those numbers down to a lower precision (like 4-bit or 8-bit) to save space.

  • 🎥 The Compression Trade-off: It’s like converting a 4K movie to 1080p. You lose a tiny bit of “crispness” (intelligence), but the file size drops by ~70%, allowing it to fit on a standard laptop.
  • 🧪 Ollama’s Sweet Spot — Q4_K_M: Ollama primarily uses 4-bit quantization (labeled Q4_K_M):
    • An 8B model shrinks from 16 GB → ~4.8 GB.
    • The loss in accuracy is usually less than 1% — imperceptible for most daily tasks.
    • Allows the model to run entirely in your computer’s RAM or GPU VRAM.

Assumes default 4-bit quantization (Q4_K_M) — what Ollama uses for standard library pulls.

Tokens, Weights & Parameters

LLM Fundamentals: Under the Hood — what a model actually is and how it thinks

LLM Fundamentals: Under the Hood

Before running models with Ollama, it helps to understand what a "model" actually is. At its simplest, an LLM is a complex piece of software trained to predict the next piece of text in a sequence — over and over, billions of times.

💡 These concepts apply to every model in the Ollama library
1 Tokens — The Currency of Language

Computers don't read words; they read numbers. Tokens are the chunks of text that a model processes as its basic unit of input and output.

  • A token isn't always a whole word. It can be a syllable, a single letter, or even a punctuation character.
  • 📏 Rule of Thumb: In English, 1,000 tokens ≈ 750 words.
  • 🔍 Example: The word "Ollama" might be split into two tokens: Oll and ama.
2 Parameters — What the "B" Means

When you see a model labeled Llama-3 8B or Phi-3 3B, the "B" stands for Billions of Parameters.

  • Definition: Parameters are internal variables that the model adjusts during training to improve its predictions.
  • Think of parameters like the connections between neurons in a brain — the more there are, the more complex patterns the model can recognize and reproduce.
3 Weights — The Strength of Connections

Each parameter has a Weight: a numerical value that determines how much influence one piece of data has on the next prediction.

  • During training, if the model correctly predicts a word it strengthens those weights. If it's wrong, it adjusts them. This process repeats trillions of times.
  • 📦 When you ollama pull a model, you are downloading a massive file of these saved, pre-trained weights. That's the entire "brain" of the model.
4 What Does "3B Parameters" Actually Mean?
Size Intelligence & Knowledge Speed & Hardware
3B Small Great for simple tasks, summarizing, and basic chat. May “hallucinate” more often. Very fast. Runs on a standard laptop or high-end phone.
8B Medium The “sweet spot” for most users. Good reasoning and follows instructions well. Needs a decent GPU (8 GB+ VRAM) to run smoothly.
70B Large High-level reasoning, coding, and deep knowledge. Very slow without professional-grade hardware (dual GPUs).
5 Inference — The Act of Thinking

When you type a prompt into Ollama and it generates a response, that process is called Inference.

The model is doing this in real time:

  • Takes your input and tokenizes it into numbers.
  • Passes those numbers through its billions of weighted parameters.
  • Calculates the most statistically likely next token.
  • Repeats that calculation one token at a time until the response is complete.

Every word you see appear is the result of that entire calculation happening in milliseconds. The larger the model, the more calculations per token — which is why bigger models are slower.

Small LLMs Compared

Several popular models under 7B parameters are available in the Ollama library, each balancing size, speed, and specialized capabilities. These "small" models are designed to run efficiently on local hardware while still performing tasks like coding, reasoning, and summarization.

📊 All models below run locally via Ollama — no cloud required
★ Side-by-Side Comparison
Model Parameters Primary Strengths Notable Weaknesses
Llama 3.2 1B, 3B Extremely fast; excellent for mobile/edge use; strong at summarization and tool use. Reasoning trails 8B+ models; struggles with high-density, long-context RAG.
Phi-3.5 Mini 3.8B Massive 128K context window; state-of-the-art reasoning for its size. Primarily English-focused; coding mostly limited to Python.
Qwen 2.5 0.5B – 7B Dominates coding and math benchmarks; strong multilingual support. 7B variants may require 8 GB+ RAM — more than ultra-light peers.

1 Context & Document Handling

How much text can the model see at once — this determines whether it can work with long documents or only short prompts.

  • 🏆 Phi-3.5 Mini is the clear leader here — its 128K token context window means it can process book-length documents in a single call. Perfect for RAG pipelines and long-form analysis.
  • ⚠️ Gemma 2 2B is significantly more limited with a default 8K token context, making it less suitable for long-form Retrieval-Augmented Generation (RAG). Use it for short, focused tasks instead.
2 Coding & Technical Tasks

Which model should you pick when generating code, writing SQL, or solving math problems?

  • 🏆 Qwen 2.5 (especially Coder variants) outperforms nearly all other sub-7B models on code generation and mathematical reasoning benchmarks. First choice for technical work.
  • ⚠️ Llama 3.2 1B/3B can handle basic text-to-SQL or simple scripts but is better suited for general assistant tasks than deep technical development.
3 Speed vs. Reasoning Quality

There is an inherent trade-off: faster models answer quickly but reason less deeply; slower models think harder and produce more accurate results.

  • ⚡ Llama 3.2 1B offers the fastest response times of the group — often generating text faster than you can read it. Ideal when speed matters more than depth.
  • 🧠 DeepSeek R1 (1.5B) uses "thinking tokens" — it explicitly works through its reasoning step-by-step before answering. This makes it more reliable for complex logic problems, but noticeably slower to produce a final answer.
4 Safety & Reliability

How well does the model resist harmful prompts, avoid hallucinations, and stay on topic?

  • 🛡️ Gemma 2 2B is built with rigorous safety filtering from Google. It is the top choice for applications requiring strict content moderation or safety policies — low toxicity and high instruction-following discipline.
  • 🛡️ Llama 3.2 also shows strong resilience against malicious prompts (96–98% safe in standard benchmarks) while maintaining its speed advantage — making it a well-rounded choice when both safety and performance are needed.

Web UI Code Flow

How a message travels from your keyboard to Ollama and back — step by step

Web UI Code Flow

When you type a message and hit Send, several things happen in sequence inside index.html. This page walks you through every step, with the actual code from the file so you can follow along.

📄 All code lives inside the <script> tag → send() function
1 Capturing the User Input

The process starts the moment you click Send or press Enter.

  • Extraction — grabs the text from the <textarea id="prompt"> element.
  • Validation — if the box is empty, the AI is already busy, or no model is selected, it stops immediately.
  • UI Update — calls appendMsg('user', text) to show your bubble in the chat, then clears the input box.
const text = input.value.trim();
if (!text || busy || !activeModel) return;

appendMsg('user', text); // Show your message in the chat
input.value = '';         // Clear the input box
2 Sending the Prompt to Ollama (The API Request)

Your prompt is sent to the local Ollama server using the browser's built-in fetch API.

  • Endpoint — targets /api/generate on the local server.
  • Method — uses an HTTP POST request.
  • Payload — JSON.stringify() converts a JS object into a JSON string containing:
    • model — the currently selected model (e.g., llama3.2:1b).
    • prompt — the text you typed.
    • stream: false — wait for the complete answer before sending anything back (no word-by-word streaming).
const res = await fetch('/api/generate', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    model: activeModel,  // e.g. "llama3.2:1b"
    prompt: text,        // what you typed
    stream: false        // wait for the full reply
  })
});
3 Receiving and Parsing the JSON Response

Once Ollama finishes generating, it sends the result back as a JSON object.

  • Parsing — await res.json() converts the raw HTTP response body into a usable JavaScript object called data.
  • Accessing the text — the Ollama API stores the generated answer in a key named response, so the code reads data.response.
  • The spinner ("Thinking…") is removed as soon as the data arrives.
const data = await res.json();

spinnerEl.remove();  // Kill the "Thinking..." animation

// data.response holds the AI's answer
appendMsg('bot', data.response || '(no response)');
4 Rendering the Response as Formatted HTML

The text isn't injected raw — it gets converted from Markdown to rich HTML first.

  • marked.js — the page loads the marked.js library via CDN. It understands Markdown syntax (bold, code blocks, lists, headers).
  • Conversion — marked.parse(text) turns the AI's Markdown string into proper HTML tags.
  • Injection — the HTML is inserted into the chat bubble using .innerHTML, so code blocks, bold text, and lists render visually. 
// Inside appendMsg() — called with role='bot'
const bubble = document.createElement('div');
bubble.className = 'bubble';

if (role === 'bot') {
  // marked.parse() converts Markdown → HTML
  bubble.innerHTML = marked.parse(text);
} else {
  bubble.textContent = text; // user messages stay as plain text
}

Ollama API Quick Reference

All endpoints with plain-language descriptions & ready-to-run curl examples

Ollama API Reference Guide

Every request targets the Ollama server running locally. By default it listens on port 11434. All POST bodies are JSON. Use the Copy button on any block to grab the exact curl command.

Base URL: http://localhost:11434
1 Generation & Chat — Talking to the Model
POST
/api/generate
Single-turn text completion. Send a prompt, get a response. This is the simplest way to ask the model a question — perfect for one-off tasks where you don't need to maintain a conversation history.
curl http://localhost:11434/api/generate -d '{
  "model": "llama3.2:1b",
  "prompt": "Summarize the risks of CVE-2025-55182.",
  "stream": false
}'
POST
/api/chat
Multi-turn chat with message history. You send an array of messages (system, user, assistant) and the model replies in context. This is the right endpoint for building anything that feels like a real conversation or an interactive agent.
curl http://localhost:11434/api/chat -d '{
  "model": "qwen2.5:1.5b",
  "messages": [
    { "role": "system", "content": "You are a cybersecurity mentor." },
    { "role": "user",   "content": "Help me with a CTF challenge." }
  ],
  "stream": false
}'
2 Model Memory & VRAM Management
GET
/api/ps
Lists every model currently living in RAM or VRAM. Use this to see what's hot in memory, how long it'll stay loaded, and how much memory it's consuming — before you run out of headroom.
curl http://localhost:11434/api/ps
ACTION
Unload / Eject a Model
Set keep_alive to 0 via the generate endpoint to immediately evict a model from memory. This is the fastest way to free up RAM or VRAM without restarting anything.
curl http://localhost:11434/api/generate -d '{
  "model": "llama3.2:1b",
  "keep_alive": 0
}'
ACTION
Preload / Warm Up a Model
Set keep_alive to -1 (no prompt needed) to force-load a model into memory without running any inference. The model will stay loaded indefinitely, so the very first real request responds instantly.
curl http://localhost:11434/api/generate -d '{
  "model": "llama3.2:1b",
  "keep_alive": -1
}'
3 Library & Disk Management
GET
/api/tags
Returns a list of all models saved to disk (in your Docker volume or local Ollama store). This tells you what's available to load — not necessarily what's in memory right now. Compare with /api/ps to see which ones are currently active.
curl http://localhost:11434/api/tags
POST
/api/pull
Downloads a model from the Ollama model library to your local storage. This is exactly what the Model Manager UI does behind the scenes. Add "stream": true to watch live download progress.
curl http://localhost:11434/api/pull -d '{
  "model": "deepseek-r1:1.5b"
}'
POST
/api/show
Returns the full metadata for a specific model: its Modelfile, system prompt, parameter count, quantization level, template, and license. Useful when you need to inspect exactly how a model is configured before using it.
curl http://localhost:11434/api/show -d '{ "name": "llama3.2:1b" }'
DELETE
/api/delete
Permanently removes a model from local storage, freeing disk space. This cannot be undone — you'd have to re-download the model. Use -X DELETE with curl because this endpoint uses the HTTP DELETE method. 
curl -X DELETE http://localhost:11434/api/delete -d '{ "name": "stablelm2:1.6b" }'
4 Advanced Utilities
POST
/api/embed
Converts text into a numerical vector (embedding). Embeddings capture the meaning of text in a way that machines can compare and search. This is the foundation of RAG (Retrieval-Augmented Generation) and vector database workflows — the model "understands" your document so you can search it by concept instead of keyword. 
curl http://localhost:11434/api/embed -d '{
  "model": "llama3.2:1b",
  "input": "Technical lab documentation for CyberLessons101"
}'
POST
/api/create
Builds a brand-new custom model from a Modelfile string — no file on disk required. You can bake in a permanent system prompt, change the base model, or alter generation parameters. The result is a standalone model you can reference by name just like any other.
curl http://localhost:11434/api/create -d '{
  "name": "iCSI-Assistant",
  "modelfile": "FROM llama3.2:1b\nSYSTEM You are an expert in blockchain security."
}'
